# 06｜样式技术选型决策（CSS Token vs TailwindCSS）

## 目标

在保持 Week2 统一 UI/UX 标准的前提下，明确后续开发阶段的样式实现路径，并评估是否引入 `tailwindcss`。

## 当前基线

- 现状：项目使用 `src/index.css` 全局样式。
- 优点：门槛低、改动直接。
- 问题：页面增多后，样式一致性与复用性容易下降。

## 方案对比

### 方案 A：继续 `index.css + design token`

优点：

- 不新增依赖，成本最低。
- 对当前代码侵入小。

缺点：

- 规范靠人工执行，一致性风险高。
- 长期维护容易出现重复样式和命名漂移。

适用：

- 页面数量少、迭代慢、团队样式纪律很强。

### 方案 B：引入 `tailwindcss`（推荐）

优点：

- 通过原子类提升开发效率，减少“写一堆新 class”。
- 配合 `tailwind.config` 可将 token 固化，强制一致性。
- 对响应式、状态样式（hover/focus/disabled）表达更直接。

缺点：

- 需要一次性迁移学习成本。
- 组件初期会出现较长 class 字符串，需要约束组织方式。

适用：

- 页面规模持续增长、需要快速且稳定地保持一致视觉。

## 推荐结论

推荐采用 **方案 B（TailwindCSS）**，但按“低风险渐进迁移”执行：

1. 先保留 `index.css`，只把 token 映射进 `tailwind.config`。
2. 新页面（例如“我的”）优先使用 Tailwind。
3. 已有页面按模块逐步迁移，不一次性重写。
4. 最后收敛冗余全局样式，保留少量 reset 和基础变量。

## 实际执行结果（Week2）

- 已完成 Tailwind 渐进接入并用于主要页面重构。
- 当前项目实际版本为 `tailwindcss@3.x`，配合 `postcss` 和 `autoprefixer` 使用。
- `src/index.css` 已缩减为 `@tailwind base/components/utilities` + 少量全局基础样式。
- 主要页面与基础组件已切换到 Tailwind class 组织。

## 渐进迁移步骤（执行时）

> 本文档仅给方案，不在本周执行依赖安装。

1. 安装依赖：`tailwindcss`、`postcss`、`autoprefixer`。
2. 初始化配置并映射 token（颜色、字体、间距、圆角、阴影）。
3. 建立约束：
  - 禁止页面级硬编码色值。
  - 组件统一使用语义化 token。
4. 从基础组件开始迁移：Button/Input/Card/Menu。
5. 再迁移页面：`我的` -> 测评作答 -> 列表与详情页。
6. 通过 UI 走查和 lint 规则收敛遗留样式。

## 风险与控制

- 风险：类名冗长影响可读性。  
控制：组件内分层、提取复用片段、必要时用 `@apply`（谨慎）。
- 风险：迁移期样式混用造成视觉偏差。  
控制：先迁基础组件，再迁页面；每次迁移都对照 token。

## 与本周目标的关系

- 用户友好：交互状态更统一，反馈更清晰。
- 降低理解成本：设计 token 通过配置固化，减少“例外”。
- 建立信任感：跨页面一致性更容易长期保持。
